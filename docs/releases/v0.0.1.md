# SDAD Inspector 0.0.1

This is the first regular GitHub release of SDAD Inspector. It gathers the
active SDAD packet, SPEC, exact current TODO, findings, owner gates, routed
documents, official control loop, and bounded Git evidence in one Korean/English
read-only interface.

> **Unsigned portable release:** these archives are not code-signed or
> notarized and are not installers. Your operating system may warn or block
> them. Verify `SHA256SUMS`, inspect the public source, and use them only if this
> limitation is acceptable in your environment.

## What is included

- A version-neutral one-line SDAD Inspector product banner in the app Overview
  and README, plus the established logo in the command bar, favicon, native
  window, and package.
- One Windows x64 archive, one macOS arm64 archive, and one Linux x64 archive
  built from the exact `v0.0.1` tag on GitHub-hosted runners.
- Exactly one **single portable executable** in each platform archive, built
  with CPython 3.12. A destination computer does not need a separate Python
  installation or an adjacent `_internal` directory.
- Windows product metadata for `SDAD Inspector 0.0.1`, a CI comparison between
  the embedded executable icon and the source `.ico`, and a shell icon-cache
  refresh after verified automatic replacement.
- The authenticated bundled official SDAD Protocol `v3.2.2` runtime, bounded
  Doctor integration, responsive React UI, Rule 5 proposal export, and redacted
  static-report support.
- A fail-closed `ProtocolAdapter` boundary between Inspector orchestration and
  SDAD engine/rule interpretation. Snapshot schema 2 records the selected
  adapter, capabilities, engine label, source paths, and bounded version lanes;
  the portable build ships only `official-sdad-3`.
- The official `Plan → Route → Implement → Verify → Report` rail, with
  Gate/Handoff kept conditional and an optional explicit `[current]` plus
  `[phase:…]` TODO marker for showing actual declared packet work without
  inferring from Git, time, ordering, or counts.
- Active packet context, exact current TODO, and bounded live Markdown evidence
  documents that open inside the existing read-only viewer.
- Project-scoped Git activity for nested repositories, safe Markdown image
  fallbacks, readable narrow layouts, collision-safe command bar, owner-gate
  rows and resizable repository labels, and a dependency-audited Vite bundle.
- `SHA256SUMS`, GitHub artifact attestations, and a frozen-only **automatic product update**
  flow for the Inspector executable.

The tagged workflow builds and directly smoke-launches each app, packages it,
then gives the archive to a separate clean hosted runner for a second launch.
Only after all three portable smokes pass does the workflow create a draft,
attest and upload all assets, and publish the regular release. Repository
immutable-release mode prevents later tag or asset replacement.

## Automatic product update boundary

The packaged app checks the canonical `LiveTrack-X/sdad-inspector` releases at
startup and every six hours. It accepts only a strictly newer published
immutable release, the exact OS/architecture asset, GitHub's SHA-256 digest,
bounded GitHub redirects and sizes, and exactly one expected regular executable
inside the archive.

After background download and verification, the UI shows a ten-second restart
countdown when no inspection is active. A copied new executable runs as a
separate helper, waits for the current process, retains one `.previous` backup,
atomically replaces the original portable executable, refreshes the Windows
shell icon cache when applicable, and relaunches the same project. The user can
postpone the restart for the current session. A failed replacement restores and
relaunches the previous executable when possible and blocks automatic retry
loops until the user explicitly retries.

This updates only the Inspector product. It does not download an SDAD engine or
write/migrate the inspected project. Source and loopback-browser modes cannot
self-update.

## Run it

1. Download `SHA256SUMS` and the one archive matching your OS and architecture.
2. Verify the archive hash using the commands in the README.
3. Extract the archive and run its only file: `SDAD-Inspector.exe` on Windows or
   `SDAD-Inspector` on macOS/Linux.
4. Pass an SDAD project path, or launch without a path and choose the folder
   containing `sdad-state.yaml`.

## Compatible SDAD projects

The primary baseline is
[SDAD Protocol v3.2.2](https://github.com/LiveTrack-X/spec-driven-ai-development/tree/v3.2.2).
The selected repository must have root `sdad-state.yaml`. The frozen
compatibility corpus covers Doctor `3.2.1` and `3.2.2`, report schemas 1 and 2,
state schemas 1 and 2, and Inspector snapshot schema 2; this app always selects
the built-in `official-sdad-3` adapter and runs its bundled authenticated `3.2.2`
engine. The adapter seam is an extension contract, not a claim that unbundled
SDAD variants are already supported.

## Important limitations

- A regular GitHub Release is not a signing or broad stable-support claim.
  Hosted-runner evidence applies to these exact unsigned archives.
- There is no installer, code signing, notarization, package registry, or
  production-readiness claim.
- The updater is an unsigned portable self-replacement path, not a signed
  installer upgrade or uninstall system. It keeps one backup but does not
  promise recovery from every filesystem, policy, power-loss, or
  endpoint-security failure.
- Inspector never executes project-declared validation commands and does not
  edit the inspected project. Rule 5 export writes only an inactive proposal to
  a separate owner-selected path after explicit confirmation.
- The one-file runtime extracts into the OS temporary directory while running;
  Linux `noexec` temporary filesystems are outside this evidence.
- Python is embedded, but platform browser/display components are not: Windows
  needs WebView2; Linux needs a working desktop and EGL/GL/XCB runtime libraries.
